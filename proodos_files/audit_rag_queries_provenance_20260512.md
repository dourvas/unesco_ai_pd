# Pre-flight audit — `rag_queries` provenance for C.3 backfill

**Date:** 2026-05-12
**Scope:** establish the empirical basis for the backfill `model_name` strategy in the C.3 piece (machine-readable AI content markers, Article 50(2)). Required by CP-1 of the C.3 design proposal review.
**Author:** Claude Code session
**Result:** **Single-constant backfill strategy is safe.** All historical AI artefacts can be marked `model_name='gemini-2.5-flash'` with no risk of mis-attribution.

---

## 1. Why this audit

The C.3 design proposal §3 D5a (retroactive backfill via management command) marks every historical AI artefact with `model_name='gemini-2.5-flash'`. The §10 R2 risk noted that if any rows were generated under the `gemini-1.5-flash` fallback branch (rag_query_system.py:262, 413), or under an earlier model standard, the constant would be wrong for those rows.

This audit answers three questions:

  1. **When did the 2.5-flash standard land in the codebase?** (git history)
  2. **Did any historical row get generated by the 1.5-flash fallback path?** (live DB)
  3. **How many rows are in scope, and who owns them?** (live DB)

---

## 2. Methodology

  - **Git:** `git log --all -S "gemini-2.5-flash" -- rag_query_system.py` and `git blame --line-porcelain rag_query_system.py | grep gemini`.
  - **Live DB:** Django shell against the dev DB (`unesco_ai_teacher_pd`, localhost), queries via raw cursor (rag_queries is outside the ORM).
  - **Proxy for fallback-path detection:** the fallback at lines 262 / 413 invokes `genai.GenerativeModel('gemini-1.5-flash')` inside `except` blocks that fire if the 2.5-flash `client.models.generate_content(...)` call raises. Rows produced by the fallback would still write to `rag_queries` via the same `INSERT INTO rag_queries (...)` SQL at line 445 — but they would NOT have `generation_tokens` populated from the new-API response, because the fallback uses the legacy `GenerativeModel` path which goes through a different response object. Proxy query: `COUNT(*) FROM rag_queries WHERE generation_tokens IS NULL OR generation_tokens = 0`. This is a *necessary* (not sufficient) condition for fallback firings to have left a trace.

## 3. Findings

### 3.1 Git history

```
$ git log --all -S "gemini-2.5-flash" -- rag_query_system.py
6b090fa Initial baseline: PROODOS EduAI platform (M1-M15)
88024e3 tmp: snapshot before baseline split
```

Only two commits in the repo's history touch the `gemini-2.5-flash` literal — the initial baseline (`6b090fa`, author-time 2026-04-23 04:38:52 UTC) and a `tmp: snapshot before baseline split` (`88024e3`). Every line of `rag_query_system.py` that names a Gemini model is blame-attributed to the initial baseline.

**Conclusion:** there has been no model-standardisation transition during the repo's recorded history. The codebase has used 2.5-flash as its default since day one of the tracked history; the 1.5-flash references are dead-fallback exception handlers, never the primary path.

Zero rows have null or zero `generation_tokens`. The 1.5-flash fallback branch has never fired in production (or, more precisely, has never written a row to `rag_queries`).

## 4. Backfill strategy decision

**Strategy: single-constant.** Backfill writes `model_name='gemini-2.5-flash'` for every historical row across all five AI-artefact kinds (rag_query, dtp_narrative, rag_feedback, peer_synthesis, rtm_position). Justified by:

  - No model-standardisation transition in git history (§3.1).
  - Zero rows in the fallback-path proxy bucket (§3.4).
  - All 98 historical rag_queries rows are within the 2.5-flash era.
  - The DTP / RAG feedback / peer synthesis / RTM tables (Django-managed) only acquired their AI-write paths after Phase A Tier 3, which post-dates the 2.5-flash standardisation.

**Orphan handling:** the 74 orphan `rag_queries` rows are **excluded** from the backfill. The `AIArtefactProvenance.user_id` FK requires a valid `auth_user`; orphans cannot legally take a provenance row. They remain unmarked in the DB but are unreachable from any user-facing surface, so this leaves no Article 50(2) gap.

**Rejected alternatives:**

  - **Split-by-cutoff** (`if created_at < X: 'gemini-1.5-flash' else: 'gemini-2.5-flash'`): no cutoff to defend — the git history shows no transition.
  - **Heuristic-by-tokens** (mark as `'unknown'` if `generation_tokens IS NULL OR = 0`): proxy returns zero matches, so the heuristic would do no work; deferred indefinitely.
  - **'unknown' fallback for all historical rows:** would weaken the export's compliance footprint without justification — the historical data IS reliably 2.5-flash.

## 5. Deployment-order implication

The C.3 design proposal §9 originally placed the backfill "between CP-11 wipe and pilot recruitment" (CP-2 of the review). Audit confirms this is wrong:

  - CP-11 wipe cascade-clears `rag_queries` rows for the 4 non-staff users (19 rows go to zero).
  - Staff rows (5 rows across 2 users) and orphan rows (74 rows) survive CP-11.
  - After CP-11, the backfill's working set is **5 staff rows + 14 DTP narratives + 22 RAG feedback + 18 peer syntheses + 34 RTM tensions** (the Django-managed counts are also from the audit but only for non-empty fields).
  - Running the backfill BEFORE CP-11 ensures staff testing of the export (in the window between C.3 deploy and CP-11) sees consistent provenance.

**Corrected deployment order:**

  1. C.3 deploy (commits 1 + 2a + 2b + 3 + migration 0005)
  2. `python manage.py backfill_ai_provenance --commit` (with typed YES confirmation)
  3. CP-11 wipe (`scripts/cp11_wipe_test_users.py --commit`)
  4. Pilot recruitment

The backfill is idempotent (get_or_create semantics, per CP-7), so any subsequent rerun is a no-op. This is the order codified in the updated §9 of the C.3 design proposal.

## 6. Followups outside the C.3 scope

  - **Orphan rag_queries cleanup** — 74 rows of detached telemetry could be deleted via a one-off `DELETE FROM rag_queries WHERE user_id NOT IN (SELECT id FROM auth_user)`. Not blocking; consider as a Phase H housekeeping task alongside TD-016. Adding here as a tentative TD entry idea (not yet opened): "TD-018 — Orphan rag_queries cleanup".
  - **Fallback-path detection robustness** — if the 1.5-flash branch ever does fire and writes a row, the proxy would still flag `generation_tokens IS NULL OR = 0` going forward. Capture this in a check in the backfill command's `--dry-run` output as an early-warning signal: "Detected N rows that may have been produced by the fallback path; review before commit."
